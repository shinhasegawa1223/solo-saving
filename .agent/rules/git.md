# Git運用ルール

本プロジェクトでは **GitHub Flow** を採用し、コミットメッセージは **Conventional Commits** に準拠します。

## 1. ブランチ戦略 (GitHub Flow)

- **main**: 常にデプロイ可能な状態を保つブランチ。
- **Feature Branches**: 新機能開発やバグ修正を行うブランチ。main から作成し、作業完了後は main へプルリクエスト（PR）を送ります。

### ブランチ名の命名規則
- feat/機能名 (例: feat/user-auth)
- fix/バグ名 (例: fix/login-error)
- docs/ドキュメント名 (例: docs/update-readme)

## 2. コミットメッセージ規則

コミットメッセージは以下の形式で記述してください（1行書き推奨）。
複数行が必要な場合複数行でも可能、コミットは日本語で行うこと

```text
プレフィックス: 内容
```

### プレフィックス一覧

| プレフィックス | 意味・用途 | 例 |
| :--- | :--- | :--- |
| **feat** | 新機能の追加 | feat: ユーザープロフィール編集機能を追加 |
| **fix** | バグ修正 | fix: フォーム送信時のエラーを修正 |
| **docs** | ドキュメントのみの変更 | docs: READMEにインストール手順を追記 |
| **style** | コードの意味に影響しない変更（改行、整形など） | style: インデントを修正 |
| **refactor** | バグ修正や機能追加ではないコードの整理 | refactor: 重複していた関数を共通化 |
| **test** | テストコードの追加・修正 | test: ログイン処理のテストを追加 |
| **chore** | ビルドプロセスやライブラリの更新など | chore: 依存ライブラリのアップデート |

## 3. プルリクエスト (PR) フロー

1. 作業ブランチを main から作成。
2. 変更をコミット（上記の規則に従う）。
3. GitHub 上で PR を作成。
4. レビュー承認後、main にマージ。
5. 作業ブランチを削除。
